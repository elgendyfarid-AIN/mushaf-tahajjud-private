# مصحف التهجد والقيام — المرحلة 0 (الأساس)

هذا المستودع هو **نقطة الانطلاق** فقط. الهدف منه إثبات أن خط الإنتاج الكامل
يعمل من أوله لآخره:

```
تحرير الكود على الهاتف (AndroidIDE)
        │
        ▼
رفع الملفات على GitHub (من كمبيوتر العمل، سحب وإفلات)
        │
        ▼
GitHub Actions يبني المشروع تلقائيًا
        │
        ▼
تحميل ملف APK الجاهز من تبويب Actions وتثبيته على الهاتف
```

التطبيق في هذه المرحلة **لا يحتوي بعد** على نص القرآن الحقيقي، ولا الصوتيات،
ولا التفاسير. هو فقط هيكل يعمل وقابل للتثبيت، وفيه:

- شريط علوي باسم التطبيق.
- زر تبديل بين "المصحف العادي" (604 صفحة) و"مصحف التهجد" (240 صفحة).
- تنقّل أفقي بالسحب بين الصفحات (يمين/يسار) مع عدّاد صفحات حقيقي.
- دعم كامل للاتجاه من اليمين لليسار (RTL).

كل هذا عمدًا، عشان نتأكد إن "الأنبوب" (pipeline) سليم 100% قبل ما نحمّله
بالميزات الثقيلة (نص القرآن، صوت، تفاسير...) واللي لو فيها أي مشكلة بناء،
هتكون أصعب تشخيصها من فوق هيكل غير مجرّب.

---

## 1) خطوات الرفع والتشغيل (من كمبيوتر العمل، متصفح فقط)

1. افتح github.com وسجّل دخول، واعمل **مستودع جديد (New repository)**
   باسم مثلاً `mushaf-tahajjud` (Private أو Public، حسب رغبتك).
2. افتح المستودع، اضغط **Add file → Upload files**.
3. فُك ضغط الملف الذي أرسلته لك، واسحب **كل المحتويات** (المجلدات والملفات
   كلها، بما فيها `.github` و`app` وملفات `build.gradle.kts`) وارفعها.
   > مهم: لازم تتأكد إن المتصفح حافظ على أسماء المجلدات (`.github/workflows/...`،
   > `app/src/main/...`). جيت هَب بيدعم رفع مجلدات كاملة بالسحب من سطح المكتب
   > في أغلب المتصفحات الحديثة.
4. اكتب رسالة Commit (مثلاً "المرحلة 0") واضغط **Commit changes** على فرع `main`.
5. روح لتبويب **Actions** فوق في المستودع. هتلاقي عملية بناء (`Build APK`)
   بدأت تلقائيًا. انتظر لحد ما تخلص (علامة ✔ خضراء)، بياخد دقيقتين لـ5 دقايق
   تقريبًا.
6. اضغط على عملية البناء، وفي الأسفل هتلاقي **Artifacts** → حمّل
   `mushaf-tahajjud-debug-apk` (هيجي كملف zip فيه الـ APK بداخله).
7. حمّل الملف، وانقله للهاتف (عبر أي طريقة: رابط مباشر لو فاتح المتصفح
   على الهاتف نفسه، أو نقل عبر السحابة).
8. على الهاتف: فُك الـ zip، وثبّت ملف الـ APK (هيطلب منك تفعيل "تثبيت من
   مصادر غير معروفة" لمتصفحك/مدير الملفات — وافق مرة واحدة وكفاية).

لو حصل خطأ في خطوة البناء (علامة ✗ حمراء في Actions)، افتح السجل (log)
وانسخ لي رسالة الخطأ، وهحلها معاك فورًا.

---

## 2) ملاحظة عن مفتاح التوقيع (debug.keystore)

الملف `app/debug.keystore` **مقصود** أنه يكون مرفوع على GitHub. هو مفتاح
تجريبي غير حساس (موجود فقط لتوقيع نسخ الاختبار)، وغرضه إن كل نسخة APK تخرج
من GitHub Actions تُوقَّع **بنفس المفتاح بالضبط** في كل مرة. لو ماعملنا
كذا، كل بناء جديد كان هيطلع بمفتاح عشوائي مختلف، وهتضطر تمسح التطبيق القديم
من الهاتف قبل كل تحديث (رسالة "App not installed").

لو يومًا ما حبيت توزيع رسمي خارج نطاق التجربة الشخصية، نعمل وقتها مفتاح
إصدار (release) خاص ونخزّنه بشكل آمن كـ GitHub Secret بدل رفعه في المستودع.

## 3) خارطة الطريق الكاملة (المراحل القادمة)

هذا مشروع كبير فعلًا — أقرب لإعادة بناء تطبيق متكامل بمستوى "آيات". تقسيمه
لمراحل مش رفاهية، ده ضروري عشان كل مرحلة نتأكد إنها بُنيت وثُبّتت بنجاح
قبل ما نراكم عليها.

| المرحلة | المحتوى |
|---|---|
| **0 ✅ (هذه المرحلة)** | الهيكل + خط الإنتاج (CI) + تبديل المصحفين + تنقّل أفقي |
| **1** | دمج بيانات القرآن الحقيقية: 604 صفحة بالرسم العثماني مطابقة لمصحف المدينة (نفس تخطيط الأسطر)، صفحات مصحف التهجد (240، ربع حزب)، البحث بالسورة/الجزء/الحزب/الربع، حفظ آخر صفحة |
| **2** | التمرير العمودي (إضافة للأفقي)، التصفح الذاتي بسرعات متغيرة، العلامات المرجعية مع ملاحظات وتصنيفات، السمات الأربعة (نهاري/ليلي/حرم/تهجد)، تعدد الخطوط وتحكم الحجم |
| **3** | التلاوات الصوتية (تحميل واستماع لعدة قراء) مع تلوين النص أثناء الصوت |
| **4** | التفاسير (4 على الأقل) والترجمات النصية (10+ لغة) |
| **5** | نظام الحفظ والمراجعة (خطة يومية، بطاقات مراجعة، تكرار متباعد)، مؤشرات تقدم القراءة |

### المصادر اللي هستخدمها لبيانات القرآن (مجانية ومفتوحة، وده تقييمي
الشخصي المستقل لأفضل مصدر متاح حاليًا، وليس لأن أداة سابقة رشّحته):

- **QUL — Quranic Universal Library** (`qul.tarteel.ai`): بيانات تخطيط
  المصحف صفحة بصفحة (604 صفحة مطابقة لمصحف المدينة سطر بسطر)، خطوط QCF
  المطابقة لطباعة مجمع الملك فهد، بيانات التفاسير، الترجمات، وتوقيتات
  الصوت لمزامنة النص. هي المصدر الأساسي الذي تعتمد عليه غالبية تطبيقات
  القرآن الحديثة (Quran.com وغيره).
- **Tanzil.net**: نص عثماني موثّق ومُدقَّق، وترجمات بعدة لغات.
- **EveryAyah.com**: ملفات صوتية لكل آية بعدة قراء، تُستخدم رسميًا في
  عشرات تطبيقات القرآن.
- مصدر تفاسير مفتوح (مجموعة Tafsir API المجانية) لتوفير ابن كثير،
  السعدي، الجلالين، الميسر دون الحاجة لبناء قاعدة بيانات من الصفر.

⚠️ ملاحظة مهمة: كل هذه المصادر مجانية للاستخدام في تطبيقات إسلامية غير
ربحية، لكن عندها شروط استخدام (مثل عدم التعديل في نص القرآن، أو ذكر
المصدر). هنلتزم بيها تلقائيًا في طريقة البناء، لكن لو فكرت مستقبلًا في
نشر التطبيق تجاريًا أو على متجر، يستحق إنك تراجع شروط كل مصدر بنفسك مرة
وحدة.

## 4) القرار التقني (وسبب كل اختيار، بدون أي إشارة لأدوات سابقة)

- **Kotlin + Jetpack Compose** بدل XML تقليدي: لأنك بتكتب الكود يدويًا على
  الهاتف بدون محرر مرئي (Designer)، فملف Kotlin واحد فيه الواجهة والمنطق
  معًا أسهل في التحرير من إدارة ملفين منفصلين (XML + Kotlin) لكل شاشة.
- **AGP 8.13.0 (وليس أحدث إصدار 9.x المتاح)**: إصدار 9 فيه تغييرات جوهرية
  في طريقة دمج Kotlin مع Gradle، وده معناه مخاطرة أعلى لفشل بناء يصعب
  تشخيصه عن بُعد عبر GitHub Actions فقط بدون IDE محلي. 8.13 مستقر وموثّق
  بشكل كامل ويدعم كل ما نحتاجه.
- **GitHub Actions بدون Gradle Wrapper محلي**: بنستخدم إجراء جاهز
  (`gradle/actions/setup-gradle`) يثبّت Gradle مباشرة على السيرفر، بدل
  الاعتماد على ملف `gradlew` ثنائي يصعب توليده يدويًا على الهاتف. أنت
  مش محتاج تشغّل بناء محلي أبدًا حسب خطة العمل المتفَق عليها.
